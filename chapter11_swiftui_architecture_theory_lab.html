Chapter 11 Lab: App Architecture and SwiftUI Part II

Introduction

This lab reinforces Chapter 11 of An iOS Developer's Guide to SwiftUI by Michele Fadda. Chapter 11 shifts from building layers to judging them. You will act like a junior architect reviewing a campus app design.

There is no perfect architecture you can copy into every project. Good architects match the structure to the problem, keep things light, and know when to ask for help.

Learning Objectives

After completing this lab, you will be able to:

  • Explain why architecture should stay light for small apps
  • Spot signs of bad architecture using a simple scorecard
  • Describe Conway's Law and how team structure shapes code
  • Recognize agency problems that lead to over-engineering
  • Compare simple MVVM with heavier options like TCA
  • Use fake service implementations for previews and testing
  • Write smart questions to ask a senior developer before committing to a pattern

Key Ideas from Chapter 11

Keeping it light means packing a day bag for class, not hauling a camping trailer for a one-hour lecture. Match the gear to the trip.

Conway's Law says your code layout mirrors how your team talks. If three friends each own one feature, you will likely get three modules — whether you planned it or not.

Agency theory warns that people sometimes pick flashy tools to look impressive, not because the app needs them. A consultant paid by the hour may suggest a mansion when you asked for a shed.

Software patterns are reusable recipes. A fake service is a stunt double that lets previews and tests run without real network calls.

Part 2: Spot Bad Architecture

Covers Chapter 11: What good software architecture is and what it is not.

Study this intentionally messy sample. List at least five problems you see.

struct ContentView: View {
    @State private var events: [String] = []

    var body: some View {
        List(events, id: \.self) { event in
            Text(event)
        }
        .onAppear {
            let url = URL(string: "https://api.example.com/events")!
            URLSession.shared.dataTask(with: url) { data, _, _ in
                if let data = data,
                   let text = String(data: data, encoding: .utf8) {
                    events = text.components(separatedBy: ",")
                }
            }.resume()
        }
    }
}

Part 3: Add Fake Services Using a Software Pattern

Covers Chapter 11: The origin of software patterns.

Chapter 11 explains that patterns solve recurring problems. A common pattern in iOS is a fake or stub service that stands in for real data. Add two new types to your Chapter 10 project.

struct EmptyEventService: EventServiceProtocol {
    func fetchEvents() -> [CampusEvent] {
        []
    }
}

struct ErrorEventService: EventServiceProtocol {
    func fetchEvents() -> [CampusEvent] {
        []
    }
}

Update your ViewModel to surface an error when the list is empty after loading. A simple check is enough for this lab:

func loadEvents() {
    isLoading = true
    errorMessage = nil
    let results = eventService.fetchEvents()
    events = results
    if results.isEmpty {
        errorMessage = "No events available right now."
    }
    isLoading = false
}

Add SwiftUI previews that inject each fake service. This is how patterns support testing and design review without running the full app.

#Preview("Loaded Events") {
    EventListView(
        viewModel: EventListViewModel(eventService: MockEventService())
    )
}

#Preview("Empty State") {
    EventListView(
        viewModel: EventListViewModel(eventService: EmptyEventService())
    )
}

#Preview("Error State") {
    let viewModel = EventListViewModel(eventService: ErrorEventService())
    viewModel.loadEvents()
    return EventListView(viewModel: viewModel)
}

Run all three previews in Xcode. Screenshot each one for submission.

Troubleshooting

Preview Shows a Blank Screen

Call loadEvents() before returning the preview for the error state. Confirm EventListView reads errorMessage from the ViewModel.

Empty and Error Previews Look the Same

Give them different preview names. Make sure the error preview sets errorMessage when the service returns an empty array.

I Am Unsure Which Clean Architecture Layer to Pick

Use this shortcut. Views are UI. ViewModels are Application. Models like CampusEvent are Domain. Services that fetch data are Data.
