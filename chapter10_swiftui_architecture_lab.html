Chapter 10 Lab: App Architecture and SwiftUI Part I

Introduction

This lab reinforces Chapter 10 of An iOS Developer's Guide to SwiftUI by Michele Fadda. You will take a messy, all-in-one screen and reorganize it into clear layers. Each layer has one job. That makes your app easier to read, test, and change.

Chapter 10 does not give you one perfect template to copy everywhere. It gives you practical tools to split your code into smaller, connected pieces.

Learning Objectives

After completing this lab, you will be able to:

  • Sketch a simple three-layer app diagram (View, ViewModel, Service)
  • Apply dependency inversion with a protocol instead of a concrete class
  • Inject dependencies through an initializer, a method, and a property
  • Keep UI state in a ViewModel using ObservableObject and @Published
  • Connect a View to a ViewModel with @StateObject
  • Explain why views should not fetch their own data directly

Key Ideas from Chapter 10

Clean layers work like a campus cafeteria. The menu board (View) shows what is available. The cashier (ViewModel) takes your order and updates the display. The kitchen (Service) actually prepares the food. The menu board does not cook.

Dependency inversion is like a wall outlet. Your lamp depends on the standard plug shape, not on which power plant made the electricity. Code should depend on a protocol, not one fixed class.

Initializer injection is handing someone everything they need at the front door. Method injection is handing them a tool only when one specific task starts. Property injection is leaving a shared tool on the workbench for anyone in the room to pick up.

Chapter 10 architecture concepts and simple analogies
Concept What it does Analogy
View Displays data and captures user taps The cafeteria menu board
ViewModel Holds screen state and handles user actions The cashier who updates the order
Service / Model Loads or stores raw data The kitchen that prepares meals
Protocol Defines what a service must do without naming one class The standard wall outlet shape
Dependency inversion High-level code depends on abstractions, not details Plug into any outlet that fits the standard
Initializer injection Pass required dependencies in init Hand tools to a worker at the front door
Method injection Pass a dependency into one specific function Hand a whisk only when the recipe step needs it
Property injection Set a dependency on a property after creation Leave a shared tool on the workbench
@Published Marks ViewModel properties that refresh the UI A bell that rings when the order changes
@Observable (iOS 17+) Newer way to track state without @Published A smarter bell that only rings for the item that changed

Prerequisites

Complete Chapters 1 through 7 first. Create a new SwiftUI app named CampusEventsArchitect. Read Chapter 10 before starting. This lab uses ObservableObject, which works on iOS 15 and later.

Lab Scenario: Refactor the Campus Events Screen

You start with one file that does everything. By the end, you will have separate files for the model, service protocol, mock service, view model, and view. The app still shows a list of campus events, but the code is organized like a real project.

Lab Instructions

Part 1: Draw Your App Diagram

Covers Chapter 10: Diagrams; Clean Architecture.

On paper or in a drawing tool, sketch three boxes stacked vertically:

  1. View — EventListView shows events and a Refresh button
  2. ViewModel — EventListViewModel holds the list and loading state
  3. Service — EventServiceProtocol fetches event data

Draw arrows showing data flow. The View asks the ViewModel to load. The ViewModel asks the Service. The Service returns events. The ViewModel updates state. The View redraws. Label your diagram and save it for submission.

Part 2: Create the Model and Service Protocol

Covers Chapter 10: Dependency inversion; different ways to decouple.

Create a new Swift file named CampusEvent.swift. Keep the model as a simple struct.

import Foundation

struct CampusEvent: Identifiable, Hashable {
    let id = UUID()
    let title: String
    let location: String
    let date: Date
}

Create EventServiceProtocol.swift. The ViewModel will depend on this protocol, not on one concrete class. That is dependency inversion.

import Foundation

protocol EventServiceProtocol {
    func fetchEvents() -> [CampusEvent]
}

Create MockEventService.swift for lab data. In a real app, you might swap this for a network service later without rewriting the ViewModel.

import Foundation

struct MockEventService: EventServiceProtocol {
    func fetchEvents() -> [CampusEvent] {
        [
            CampusEvent(title: "Study Jam", location: "Library Room 204", date: .now),
            CampusEvent(title: "Club Fair", location: "Student Center", date: .now.addingTimeInterval(86400)),
            CampusEvent(title: "Basketball Game", location: "Gymnasium", date: .now.addingTimeInterval(172800))
        ]
    }
}

Part 3: Build the ViewModel with Initializer Injection

Covers Chapter 10: Initializer injection; a matter of state.

Create EventListViewModel.swift. Pass the service through init. Store it in a private constant. Publish the properties the View needs.

import Foundation

final class EventListViewModel: ObservableObject {
    @Published var events: [CampusEvent] = []
    @Published var isLoading = false
    @Published var errorMessage: String?

    private let eventService: EventServiceProtocol

    init(eventService: EventServiceProtocol) {
        self.eventService = eventService
    }

    func loadEvents() {
        isLoading = true
        errorMessage = nil
        events = eventService.fetchEvents()
        isLoading = false
    }
}

The ViewModel never creates MockEventService itself. You hand it in from outside. That makes testing and swapping implementations much easier.

Part 4: Build a View That Only Displays and Sends Actions

Covers Chapter 10: A matter of state; chopping your code into parts.

Create EventListView.swift. The View should not contain hard-coded event data. It reads from the ViewModel and calls loadEvents() when needed.

import SwiftUI

struct EventListView: View {
    @StateObject private var viewModel: EventListViewModel

    init(viewModel: EventListViewModel) {
        _viewModel = StateObject(wrappedValue: viewModel)
    }

    var body: some View {
        NavigationStack {
            Group {
                if viewModel.isLoading {
                    ProgressView("Loading events...")
                } else if let errorMessage = viewModel.errorMessage {
                    Text(errorMessage)
                        .foregroundColor(.red)
                } else {
                    List(viewModel.events) { event in
                        VStack(alignment: .leading, spacing: 4) {
                            Text(event.title)
                                .font(.headline)
                            Text(event.location)
                                .foregroundColor(.secondary)
                            Text(event.date, style: .date)
                                .font(.caption)
                        }
                    }
                }
            }
            .navigationTitle("Campus Events")
            .toolbar {
                ToolbarItem(placement: .navigationBarTrailing) {
                    Button("Refresh") {
                        viewModel.loadEvents()
                    }
                }
            }
            .onAppear {
                viewModel.loadEvents()
            }
        }
    }
}

Update ContentView.swift to wire everything together at the top level. This is sometimes called the composition root — the one place that chooses concrete implementations.

import SwiftUI

struct ContentView: View {
    var body: some View {
        EventListView(
            viewModel: EventListViewModel(eventService: MockEventService())
        )
    }
}

Run the app. You should see three events and a working Refresh button. The View file has zero knowledge of where the data came from.

Part 5: Use Method Injection for Date Formatting

Covers Chapter 10: Method injection.

Sometimes a helper only needs a tool for one job. Method injection passes that tool into a single function. Add this to a new file named EventFormatting.swift or at the bottom of EventListView.swift.

import Foundation

extension DateFormatter {
    static let eventDisplay: DateFormatter = {
        let formatter = DateFormatter()
        formatter.dateStyle = .medium
        formatter.timeStyle = .short
        return formatter
    }()
}

func formatDate(_ date: Date, using formatter: DateFormatter) -> String {
    formatter.string(from: date)
}

The list row calls formatDate(event.date, using: .eventDisplay). Replace the simple Text(event.date, style: .date) line from Part 4 with this version:

Text(formatDate(event.date, using: .eventDisplay))
    .font(.caption)

The formatter is injected into the function instead of being created inside it every time. That keeps formatting flexible and easy to test.

Part 6: Try Property Injection with environmentObject

Covers Chapter 10: Property injection; different ways to decouple.

Property injection sets a dependency on a shared property. In SwiftUI, .environmentObject is a common form. Create a simple settings object and inject it once at the app root.

import SwiftUI

final class AppSettings: ObservableObject {
    @Published var campusName = "MCC Campus"
}

Inject it in your app entry file:

@main
struct CampusEventsArchitectApp: App {
    @StateObject private var settings = AppSettings()

    var body: some Scene {
        WindowGroup {
            ContentView()
                .environmentObject(settings)
        }
    }
}

Read it inside EventListView without passing it through every initializer:

@EnvironmentObject private var settings: AppSettings

// Inside the navigation title area or a header:
Text(settings.campusName)
    .font(.subheadline)
    .foregroundColor(.secondary)

Use property injection sparingly for app-wide shared values. Prefer initializer injection for required dependencies like your event service.

Part 7: Optional Stretch — @Observable on iOS 17

Covers Chapter 10: iOS 17 changes in state management.

Chapter 10 also covers the @Observable macro introduced in iOS 17. If your deployment target is iOS 17 or later, try rewriting the ViewModel like this:

import Observation

@Observable
final class EventListViewModel {
    var events: [CampusEvent] = []
    var isLoading = false
    var errorMessage: String?

    private let eventService: EventServiceProtocol

    init(eventService: EventServiceProtocol) {
        self.eventService = eventService
    }

    func loadEvents() {
        isLoading = true
        errorMessage = nil
        events = eventService.fetchEvents()
        isLoading = false
    }
}

In the View, replace @StateObject with @State and remove the custom initializer wrapper. This is optional. The ObservableObject version is fine for this course.

Troubleshooting

List Is Empty

Confirm loadEvents() runs in .onAppear. Check that MockEventService.fetchEvents() returns at least one event. Make sure events is marked @Published.

View Does Not Update After Loading

The ViewModel must be a class conforming to ObservableObject. Use @StateObject in the View, not @ObservedObject, when the View creates the ViewModel.

Cannot Find AppSettings in EventListView

Add .environmentObject(settings) on the root view in your App file. Declare @EnvironmentObject private var settings: AppSettings in the child view.

StateObject Wrapper Error in Init

When injecting a ViewModel through init, assign it with _viewModel = StateObject(wrappedValue: viewModel). The underscore accesses the property wrapper storage.

Too Many Files Feel Overwhelming

That is normal. Each file has one clear job. Compare it to separating notes into labeled binders instead of one giant notebook.
