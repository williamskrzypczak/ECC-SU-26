Chapter 1 Lab: SwiftUI Development Environment

Introduction

You will build a simple Campus Welcome app from scratch. You will explore Xcode, SwiftUI, previews, the simulator, playgrounds, tests, and the app submission process.

Think of this lab like setting up a workshop before building furniture. You are learning where the tools live and how to test your work. You can complete this lab in about 15 minutes. You need a Mac with Xcode 15 or later installed.

Learning Objectives

After completing this lab, you will be able to:

  • Explain what SwiftUI is and how it differs from older UIKit storyboard approaches
  • Create a new Multiplatform SwiftUI project using an Xcode template
  • Identify the app entry point (@main, App, and WindowGroup)
  • Build a simple view with Text, VStack, and SwiftUI modifiers
  • Use the preview canvas and iOS simulator to check your UI
  • Run a quick playground experiment and a unit test
  • Describe the high-level steps to submit an app to the App Store

Key Ideas

SwiftUI is declarative. You describe what the screen should look like. The system figures out how to draw it. It is like ordering from a menu instead of writing every cooking step yourself.

Xcode is your garage. It holds your code editor, preview canvas, simulator, and test tools in one place. Previews are like a mirror before a show. The simulator is the full dress rehearsal.

Part 1: Create Your First Multiplatform SwiftUI Project

Topics: Exploring SwiftUI with Xcode and Creating a multi-platform SwiftUI project.

You will create a new project from scratch. Follow these steps in Xcode.

  1. Open Xcode. Choose File → New → Project.
  2. Select the Multiplatform tab at the top.
  3. Choose the App template. Click Next.
  4. Set the product name to CampusWelcome.
  5. Set Interface to SwiftUI and Language to Swift.
  6. Choose a save location. Click Create.

Look at the Project Navigator on the left. Find the Shared folder. It holds code used across iOS and macOS. Notice the test targets in the project too.

Part 2: Trace How the App Starts

Topics: Your first SwiftUI project and How the app starts.

Open CampusWelcomeApp.swift. Older iOS apps used an AppDelegate file. SwiftUI apps use a small struct marked with @main instead. The code looks like this:

import SwiftUI

@main
struct CampusWelcomeApp: App {
    var body: some Scene {
        WindowGroup {
            ContentView()
        }
    }
}

Answer these questions in your submission:

  1. What does @main tell Swift? (Hint: it marks the starting point.)
  2. What is WindowGroup? (Hint: it holds your app's view hierarchy.)
  3. Which view loads first when the app launches?

Part 3: Build Your Welcome Screen

Topics: Exploring SwiftUI with Xcode and declarative view code.

Open ContentView.swift. Replace the default code with a welcome screen. Include an app title, your name, a VStack, and at least three modifiers.

import SwiftUI

struct ContentView: View {
    var body: some View {
        VStack(spacing: 12) {
            Text("Campus Welcome")
                .font(.largeTitle)
                .foregroundColor(.blue)
            Text("Built by Your Name")
                .font(.headline)
                .foregroundColor(.secondary)
        }
        .padding()
        .border(Color.gray, width: 1)
    }
}

#Preview {
    ContentView()
}

Replace Your Name with your actual name. Build the project with Product → Build (Cmd+B). Fix any errors before moving on.

Task 3.1: Modifier Order Experiment (Stretch Task)

This lab shows that modifier order changes the result. Create two preview versions. Put .padding() before .border() in one. Reverse the order in the other.

struct PaddingFirstView: View {
    var body: some View {
        Text("Padding first")
            .padding()
            .border(Color.blue, width: 3)
    }
}

struct BorderFirstView: View {
    var body: some View {
        Text("Border first")
            .border(Color.blue, width: 3)
            .padding()
    }
}

#Preview("Padding First") {
    PaddingFirstView()
}

#Preview("Border First") {
    BorderFirstView()
}

Take screenshots of both previews. In one or two sentences, explain why they look different. Modifiers wrap the view in the order you write them, like putting on socks before shoes versus shoes before socks.

Part 4: Preview vs Simulator

Topics: Previews and the simulator.

Use both tools to check your welcome screen. They serve different purposes.

  1. In ContentView.swift, click Resume in the preview canvas (or press Option-Cmd-P).
  2. Confirm your welcome screen appears in the canvas.
  3. Select an iPhone simulator from the device menu in the toolbar.
  4. Press Run (Cmd+R). Wait for the simulator to launch.
  5. Confirm your welcome screen appears in the simulator.

A playground lets you test small code fragments without a full project. Think of it as a practice room.

Tests catch bugs before users see them. Your project already includes a test target.

Task 5.3: App Submission Overview

You will not submit an app in this lab. This section introduces the real-world process. Here are the main steps:

  1. Join the Apple Developer Program (required for App Store release).
  2. Archive your app in Xcode (Product → Archive).
  3. Upload the build to App Store Connect.
  4. Submit the app for Apple's review.
  5. Release the app after approval.

Answer this question: Why do you not need a paid developer account to use the simulator? (Hint: the simulator runs only on your Mac for learning and testing.)

Troubleshooting

Preview Canvas Not Showing

Click Editor → Canvas to show the preview pane. Press Option-Cmd-P to resume a paused preview. Make sure #Preview is at the bottom of your view file.

Build Errors in ContentView

Check that every opening brace has a closing brace. Confirm import SwiftUI is at the top. Read the error message in the Issue Navigator on the left.

Simulator Will Not Launch

Pick a different iPhone simulator from the device menu. Choose Product → Clean Build Folder (Shift-Cmd-K), then run again.

Tests Fail on a Fresh Project

A new project should pass its default tests. If tests fail, note the error in your submission. Ask your instructor for help during office hours.
