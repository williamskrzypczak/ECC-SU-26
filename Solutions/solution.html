Chapter 0 Solution: GitHub and Xcode Source Control

Xcode project name: GitPractice

This reference shows the expected Git workflow for the Chapter 0 lab. There is no Swift source code to copy. Your finished repo history is the solution.

Workflow Checklist

Expected steps and Xcode menu paths
Part Action Xcode menu path
1 Sign in to GitHub Xcode → Settings → Accounts → + → GitHub
2 Create project with Git File → New → Project (check Source Control box)
2 Connect to GitHub Source Control → New Remote Repository…
2 Initial push Source Control → Push
3 Commit code change Source Control → Commit… (Cmd+Option+C)
4 Push commit Source Control → Push
5 Pull README edit Source Control → Pull
6 Create branch Source Control → New Branch…
6 Commit on branch Source Control → Commit…
7 Switch to main Branch picker in toolbar → main
7 Merge branch Source Control → Merge…
7 Push merged main Source Control → Push

Commit Message Examples

Good vs. weak commit messages
Quality Example Why
Good Add my name to welcome screen Describes exactly what changed
Good Add course note to README Short, clear, and specific
Good Add tagline on add-tagline branch Notes both the change and the branch context
Weak update Too vague — update what?
Weak fix stuff Does not help you find this change later
Weak asdfasdf Meaningless — treat commit messages like labels on folders

Expected Final State

When you finish the lab, your GitHub repo should look like this:

  • main branch contains your name in ContentView.swift
  • main branch contains the tagline text from the merge
  • README.md includes the course note you added on GitHub.com
  • At least four commits visible in the commit history
  • Repo is public and accessible at github.com/your-username/GitPractice

Branch History Diagram

Your branches should follow this pattern:

main: ●───●───●─────────────● (initial → name → README pull → merge) \ add-tagline: ●───● (create branch → add tagline)

The README commit happens on GitHub.com and arrives on main via pull. The tagline commit happens on add-tagline and joins main via merge.

Sample Reflection Answers

What is a commit? A commit is a saved snapshot of your project with a message that describes what changed.

What is the difference between push and pull? Push sends your local commits to GitHub. Pull downloads commits from GitHub to your Mac.

Why use a branch? A branch lets you try changes safely without breaking the working code on main.

Troubleshooting Quick Reference

Common problems and fixes
Problem Fix
Push rejected Pull first, then push again
Wrong branch Use the branch picker in the toolbar to switch
Merge blocked Commit or discard uncommitted changes first
Not signed in Re-add GitHub under Settings → Accounts
No remote repo option Create empty repo on GitHub.com, then add remote URL in Xcode
